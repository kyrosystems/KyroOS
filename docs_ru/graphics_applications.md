# Написание графических приложений для KyroOS

KyroOS предоставляет гибкий подход к разработке графических приложений, начиная от прямого доступа к фреймбуферу для максимального контроля и производительности, до использования аппаратного ускорения GPU через стандартизированные API, такие как Vulkan, OpenGL и OpenCL. Этот документ описывает основные подходы и концепции для создания графических приложений и игр в KyroOS.

## 1. Прямой доступ к фреймбуферу

Самый простой и непосредственный способ отрисовки графики в KyroOS — это прямой доступ к фреймбуферу. Ядро предоставляет приложению прямой доступ к видеопамяти, что исключает накладные расходы на переключение контекста и взаимодействие с графическим сервером.

### Получение информации о фреймбуфере

Для начала приложение должно запросить информацию о текущем фреймбуфере с помощью системного вызова `SYS_GFX_GET_FB_INFO`. Этот системный вызов отобразит физические страницы фреймбуфера непосредственно в виртуальное адресное пространство вашего процесса и вернет структуру с его параметрами.

```c
#include <syscall.h> 
#include <kprintf.h> // Для kprintf, если доступно
#include <stddef.h>  // Для NULL

// Определение структуры, аналогичной gfx_framebuffer_info_t из ядра,
// для использования в пользовательском пространстве.
// Убедитесь, что она соответствует определению в src/include/gfx_driver.h или syscall.h
typedef struct {
    void* address;
    uint64_t pitch;
    uint32_t width;
    uint32_t height;
    uint8_t bpp;
    uint8_t memory_model;
} userspace_fb_info_t;

int main() {
    userspace_fb_info_t fb_info;
    
    // Вызываем системный вызов для получения информации о фреймбуфере
    // SYS_GFX_GET_FB_INFO - это константа системного вызова, определенная в src/include/syscall.h
    int ret = syscall(SYS_GFX_GET_FB_INFO, (uintptr_t)&fb_info, 0, 0, 0, 0);

    if (ret == 0) {
        kprintf("Framebuffer Info:\n");
        kprintf("  Address: 0x%p\n", fb_info.address);
        kprintf("  Resolution: %lu x %lu\n", (uint64_t)fb_info.width, (uint64_t)fb_info.height);
        kprintf("  Pitch: %lu bytes\n", fb_info.pitch);
        kprintf("  BPP: %u\n", fb_info.bpp);

        // Пример отрисовки красного пикселя в левом верхнем углу
        if (fb_info.address != NULL && fb_info.width > 0 && fb_info.height > 0) {
            uint32_t* pixel_buffer = (uint32_t*)fb_info.address;
            // Рассчитываем смещение для пикселя (0,0)
            // При bpp=32, каждый пиксель - это 4 байта (uint32_t)
            // pitch - это байты на строку, поэтому делим на sizeof(uint32_t)
            uint64_t pixel_offset = (0 * (fb_info.pitch / sizeof(uint32_t))) + 0;
            pixel_buffer[pixel_offset] = 0x00FF0000; // Красный цвет (AARRGGBB)
            kprintf("Drawn a red pixel at (0,0).\n");
        }
    } else {
        kprintf("Failed to get framebuffer info. Error: %d\n", ret);
    }

    // Приложение должно продолжать работать или ждать других событий
    // для поддержания отрисованного состояния.
    while (1) {
        // Заглушка, чтобы приложение не завершалось сразу
        syscall(SYS_YIELD, 0, 0, 0, 0, 0); 
    }

    return 0;
}
```

### Основы отрисовки

После получения адреса фреймбуфера вы можете напрямую записывать данные в эту память. Формат пикселей определяется `bpp` (bits per pixel) и `memory_model`. Для наиболее распространенных 32-битных фреймбуферов (bpp=32) пиксели обычно представлены в формате ARGB или XRGB (0xAARRGGBB), где AA — альфа-канал, RR — красный, GG — зеленый, BB — синий.

Для предотвращения визуальных артефактов (разрывов, мерцания) рекомендуется реализовать собственную двойную или тройную буферизацию в пользовательском пространстве, если ядро не предоставляет достаточно быструю `fb_flush` для вашей задачи.

## 2. Использование аппаратных API: Vulkan, OpenGL, OpenCL

KyroOS закладывает основу для использования аппаратного ускорения GPU через современные графические API. Это достигается благодаря модульной архитектуре драйверов, где ядро предоставляет общий интерфейс (`gfx_driver_interface_t`) для взаимодействия с аппаратурой GPU.

### Обзор архитектуры

1.  **Общий интерфейс драйвера (`gfx_driver.h`):** Определяет набор абстрактных функций (например, `get_capabilities`, `allocate_gpu_memory`, `submit_command_buffer`), которые должны быть реализованы каждым конкретным драйвером GPU.
2.  **Драйверы конкретных производителей (например, `amdgpu`):** Реализуют `gfx_driver_interface_t`, предоставляя низкоуровневое взаимодействие с аппаратурой GPU (MMIO, BARs, регистры, собственные форматы команд).
3.  **Пользовательские библиотеки:** В будущем будут созданы пользовательские библиотеки (например, `libvulkan.so`, `libopengl.so`, `libopencl.so`), которые будут использовать системные вызовы KyroOS для взаимодействия с активным `gfx_driver_interface_t` ядра. Эти библиотеки будут предоставлять стандартный API Vulkan/OpenGL/OpenCL для разработчиков приложений.

### Процесс разработки приложения (концептуально)

Для разработки приложения с использованием аппаратного ускорения вам потребуется:

1.  **Обнаружение GPU и запрос интерфейса:**
    *   В будущем KyroOS предоставит системный вызов для получения `gfx_driver_interface_t` активного GPU.
    *   Приложение сможет запросить возможности GPU (`get_capabilities`) для определения поддержки Vulkan, OpenGL, OpenCL.

2.  **Управление контекстом API:**
    *   Пользовательские библиотеки (Vulkan Loader, Mesa/Gallium3D для OpenGL/OpenCL) будут отвечать за создание и управление контекстами этих API.
    *   Они будут использовать `gfx_driver_interface_t` ядра для выделения GPU-памяти (`allocate_gpu_memory`), отправки команд (`submit_command_buffer`) и управления дисплеем.

3.  **Выделение GPU-памяти:**
    *   Для текстур, буферов вершин, буферов индексов и других данных, требующих размещения в видеопамяти, приложения будут использовать API соответствующих графических библиотек (Vulkan/OpenGL/OpenCL), которые, в свою очередь, будут обращаться к `allocate_gpu_memory` ядра.

4.  **Отправка команд:**
    *   Сформированные командные буферы (например, Vulkan Command Buffers, OpenGL Display Lists) будут отправляться на выполнение GPU через `submit_command_buffer` ядра.

**Важно:** Полная реализация пользовательских библиотек Vulkan, OpenGL и OpenCL — это очень сложная и объемная задача, которая будет развиваться в KyroOS постепенно. На данный момент представлена архитектурная основа в ядре.

## 3. Разработка игр в KyroOS

Разработка игр в KyroOS может использовать как прямой доступ к фреймбуферу для простых 2D-игр, так и аппаратное ускорение GPU для более сложных 2D/3D-проектов.

### Общие принципы

*   **Игровой цикл:** Каждая игра будет работать в бесконечном цикле, который включает:
    *   Обработку ввода (событий клавиатуры, мыши).
    *   Обновление состояния игры (логика, физика).
    *   Отрисовку нового кадра.
*   **Обработка ввода:**
    *   Используйте системный вызов `SYS_INPUT_POLL_EVENT` для получения событий от клавиатуры и мыши из централизованной очереди событий ядра.
    *   Приложение само должно обрабатывать эти события, например, перемещая курсор мыши, реагируя на нажатия клавиш.
*   **Отрисовка:**
    *   **2D-игры (простые):** Могут напрямую записывать пиксели во фреймбуфер.
    *   **2D/3D-игры (сложные):** Будут использовать высокоуровневые графические API (Vulkan/OpenGL) через пользовательские библиотеки для эффективной отрисовки сложной графики.
*   **Звук:** KyroOS предоставит API для воспроизведения звука, которое будет использовать аудиодрайверы ядра (например, AC97) для вывода аудиопотоков.

### Курсор мыши

Для приложений, требующих отображения курсора мыши:
1.  Ядро KyroOS предоставляет необработанные события движения мыши через `SYS_INPUT_POLL_EVENT`.
2.  Само приложение отвечает за отрисовку курсора (например, небольшого спрайта) в нужном месте фреймбуфера.
3.  При получении события `EVENT_MOUSE_MOVE`, приложение должно перерисовать свой курсор в новых координатах. Это может быть сделано путем сохранения фонового изображения, отрисовки курсора, а затем восстановления фона при следующем движении.

Например:

```c
// Примерный код обработки движения мыши в игровом цикле
void handle_mouse_move(int32_t dx, int32_t dy) {
    // Обновить положение курсора
    static int32_t mouse_x = 0;
    static int32_t mouse_y = 0;

    // Сначала стереть старый курсор (восстановить фон)
    // draw_background_at(mouse_x, mouse_y, CURSOR_WIDTH, CURSOR_HEIGHT);

    mouse_x += dx;
    mouse_y += dy;

    // Ограничить курсор пределами экрана
    if (mouse_x < 0) mouse_x = 0;
    if (mouse_y < 0) mouse_y = 0;
    // (нужны fb_info.width и fb_info.height)

    // Отрисовать новый курсор
    // draw_cursor_at(mouse_x, mouse_y, CURSOR_COLOR);
}

// В основном цикле игры:
// event_t ev;
// while (syscall(SYS_INPUT_POLL_EVENT, (uintptr_t)&ev, 0, 0, 0, 0) == 0) {
//     if (ev.type == EVENT_MOUSE_MOVE) {
//         handle_mouse_move(ev.mouse.dx, ev.mouse.dy);
//     }
//     // ... обработка других событий
// }
```

Эта архитектура предоставляет разработчикам приложений полный контроль над тем, как их графика и ввод взаимодействуют с системой, позволяя создавать как простые, так и высокопроизводительные интерактивные приложения.
